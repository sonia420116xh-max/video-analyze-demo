# AI 短视频爆款模式拆解工程汇报

## 1. 项目定位

本工程是一个短视频内容结构化分析工具。它的目标不是简单生成视频摘要，而是把一条短视频拆成可复盘、可对比、可沉淀的方法论数据：

- 判断视频属于哪类爆款表达模式。
- 按时间轴拆成多个分镜。
- 给每个分镜标注画面内容、口播内容、商品品类、叙事角色、转化作用和证据帧。
- 保存视频、分镜图片和分析结果，方便后续复盘和对比。

一句话总结：系统先把视频转换成“字幕 + 视觉关键帧 + 时间轴”，再用内置的爆款模式分类体系约束 AI 输出统一 JSON 分镜结果。

## 2. 使用流程

用户上传视频或粘贴视频链接
    ↓
FastAPI 接收 → 创建 job → 写入 SQLite (status=queued)
    ↓
如果是视频链接：yt-dlp 下载视频 → 保存任务预览视频
    ↓
后台线程开始分析 → 更新 SQLite (status=processing)
    ↓
AI 分析完成 → 结果写入 SQLite analyses 表 + 视频/帧存文件
    ↓
更新 SQLite job (status=completed)
    ↓
前端轮询 /analysis-jobs/{job_id} → 读 SQLite 返回状态

用户在前端页面上传视频，或粘贴 TikTok / Instagram / YouTube 等公开视频链接后，系统会进入后台分析任务。

完整流程如下：

1. 前端上传视频，或粘贴一个 `http://` / `https://` 开头的视频链接，并选择可用模型和拆解粒度。
2. 前端把本地文件作为 `file` 字段提交，或把视频链接作为 `video_url` 字段提交；两者只能选一种。
3. 后端创建后台任务 `job_id`。
4. 如果是本地上传，后端把文件写入任务目录；如果是视频链接，后端先用 `yt-dlp` 下载视频。
5. 链接下载成功后，后端生成任务预览视频地址，并把任务状态更新为“视频下载完成，正在抽帧、转写和分析”。
6. 后端用 MoviePy 读取视频，提取音频。
7. 后端用 Whisper 系列模型把音频转成带时间戳的字幕；如果用户手动粘贴字幕，则跳过音频提取和 ASR。
8. 后端按拆解粒度从视频中采样视觉关键帧，压缩后转成 base64 图片。
9. 后端组装分析 prompt，把字幕、关键帧和爆款分类体系一起发给大模型。
10. 大模型返回 JSON 数组，每个对象代表一个分镜。
11. 后端对模型结果做后处理，包括字段补齐、分类纠偏、抽取分镜缩略图。
12. 后端把原视频、分镜图和 JSON 结果保存到本地存储和 SQLite。
13. 前端轮询任务状态，完成后展示分镜列表和历史记录。

## 3. 技术实现

### 前端

前端使用 Vue 3、Element Plus 和 Axios。

主要职责：

- 上传视频文件。
- 粘贴视频链接并提交后台下载分析。
- 选择分析模型。
- 选择拆解粒度：粗略、均衡、精细。
- 展示后台任务状态。
- 轮询分析进度。
- 展示视频预览、爆款公式、分镜列表和历史记录。
- 点击分镜后跳转到视频对应时间点。
- 在分析详情中展示“一键复制”入口，基于已拆解视频生成用户新商品脚本。
- 点击黄金 3 秒标签时展示命中依据和复刻方法说明。

模型下拉框现在由后端接口返回，只展示“已实现且当前可用”的具体模型名。前端不会再展示 GPTProto 这类通道名；选择 `gpt-4o` 时后端实际调用 `gpt-4o`，选择 `gemini-2.5-pro` 时后端实际调用 `gemini-2.5-pro`。

“一键复制”页面采用轻量商品输入：

- 主输入只要求用户填写或上传“需要带货的商品”，并填写“希望突出的内容”。
- 前端会调用 `POST /api/product-selling-points`，让模型根据商品标题、产品品类、用户输入和可选图片生成默认卖点；用户可以继续修改。模型失败时才使用本地规则兜底。
- 物流快、客服服务态度好、产品质量好、适用人群、价格划算等卖点推荐标签可以一键追加。
- 产品品类、目标用户、使用场景、价格优惠、目标时长、生成模型、表达语气和补充要求收在“高级设置（可选）”里。
- 上传产品图片只用于本次预览和生成请求，不保存到历史记录。
- 点击生成时，前端会把用户商品推断类目和参考视频的 `product_classification` 做比较；如果类目不一致，会二次确认后再调用后端接口。

### 后端

后端使用 FastAPI。

主要职责：

- 提供视频上传接口。
- 提供视频链接分析入口，并用 `yt-dlp` 下载远端视频。
- 管理后台分析任务。
- 调用视频处理工具完成抽帧和提音频。
- 调用 Whisper 做音频转写。
- 调用大模型做分镜分析。
- 保存历史分析记录。

关键接口：

- `POST /api/analyze`：提交视频分析任务，支持上传文件 `file` 或视频链接 `video_url`。
- `GET /api/analysis-jobs/{job_id}`：查询后台任务状态。
- `GET /api/analyses`：获取历史分析列表。
- `GET /api/analyses/{analysis_id}`：获取单条分析详情。
- `POST /api/analyses/{analysis_id}/reanalyze`：重新拆解历史视频。
- `POST /api/product-selling-points`：根据用户商品信息和可选图片生成默认带货卖点。
- `POST /api/analyses/{analysis_id}/script-copy`：基于历史拆解结果和用户商品信息生成复刻脚本。
- `GET /api/model-options`：获取当前可用模型列表。

脚本复刻接口不重新分析源视频，也不写数据库。它会读取指定 `analysis_id` 的已保存分镜，必要时按 `source_model` 选择某个模型版本，把源视频的爆款公式、分镜模板、卖点角度、视觉手法、转化作用和黄金 3 秒复刻说明拼成 prompt，再和用户商品信息、可选产品图一起发给所选模型。模型返回结构化 JSON 对象，前端直接展示并支持复制。

## 4. 使用的工具和模型

### 视频处理

- yt-dlp：解析和下载 TikTok、Instagram、YouTube 等平台公开视频链接。
- MoviePy：读取视频、提取音频、按时间点抽帧。
- Pillow：压缩和保存图片。

### 语音转文字

系统按优先级尝试：

1. `faster-whisper`：本地 Whisper，优先使用。
2. `stable-whisper`：本地 Whisper 备用方案。
3. OpenAI Whisper：如果本地不可用且配置了 `OPENAI_API_KEY`，走云端转写。

如果视频没有音频，或没有可用转写能力，系统会进入“无字幕模式”，用视觉关键帧继续分析。

### 大模型分析

目前支持的分析入口：

- `gemini-2.5-pro`：通过 GPTProto `/v1/chat/completions` 调用，项目内置默认 GPTProto key，也可以用环境变量覆盖。
- `gpt-4o`：通过 GPTProto `/v1/responses` 调用，使用 `input_text + input_image` 传入字幕与关键帧。
- `claude-sonnet-4-5-20250929`：通过 GPTProto `/v1/chat/completions` 调用，使用 openai-format File Analysis 传入关键帧。
- `qwen-turbo`：通过 DashScope 文本生成接口调用，需要 `DASHSCOPE_KEY`，未配置时前端不展示。

文心一言目前前端已不展示，因为后端还没有完整实现对应分析分支。后续如果要接入，只需要补齐后端调用逻辑和 key 判断。

### 数据存储

- SQLite：保存分析记录、模型、视频路径、分镜 JSON、爆款分类、产品分类、创建时间等。
- 本地文件系统：
  - `storage/videos/` 保存原视频。
  - `storage/frames/{analysis_id}/{model}/` 保存每条分析在对应模型下的分镜图。
  - `storage/jobs/{job_id}/` 保存后台任务的中间视频文件，包含上传文件或从 URL 下载的视频。
  - `storage/analysis.db` 保存历史记录。

## 5. 视频 URL 分析方式

视频 URL 分析复用同一个 `POST /api/analyze` 接口，但提交的是 `video_url` 表单字段，而不是 `file`。

### 前端规则

前端输入框支持粘贴 TikTok / Instagram / YouTube 等视频链接。提交前会做三类校验：

- 必须上传视频或填写视频链接，不能两者都为空。
- 上传文件和视频链接只能二选一，不能同时提交。
- 视频链接必须以 `http://` 或 `https://` 开头。

提交时前端会固定带上：

```text
video_url=<用户粘贴的视频链接>
model=<当前模型>
async_mode=true
breakdown_granularity=<coarse|balanced|fine>
transcript_override=<可选手动字幕>
```

URL 分析一定走后台任务。后端会立即返回 `job_id`，前端随后轮询 `/api/analysis-jobs/{job_id}`。如果链接下载失败，前端会提示用户把视频下载到本地后再用“上传视频”手动解析。

### 后端下载流程

后端收到 `video_url` 后，会进入 `start_url_analysis_job()`：

```text
收到 video_url
→ 用链接路径猜测一个文件名，创建 analysis_jobs 记录
→ 创建 storage/jobs/{job_id}/ 任务目录
→ 后台线程开始下载
→ yt-dlp 下载视频到任务目录
→ persist_job_preview_video() 生成可预览的视频地址
→ 更新 job.video_url 和进度文案
→ 调用 run_analysis_job() 进入常规抽帧、ASR、AI 分析流程
```

下载函数 `download_video_from_url()` 的关键配置：

- `format` 使用 `bv*+ba/b[ext=mp4]/b`，优先下载视频轨和音频轨并合并，退回到可用 mp4 或最佳单文件。
- `merge_output_format` 固定为 `mp4`。
- `noplaylist=true`，避免误下载整个列表。
- `socket_timeout=30`，`retries=1`，避免后台任务长时间卡住。

下载完成后，后端会从 `yt-dlp` 返回信息和任务目录中寻找实际视频文件，并用安全文件名进入后续分析。后续的抽帧、音频转写、AI prompt、分镜缩略图和入库逻辑与本地上传完全一致。

### 下载失败处理

如果链接不合法、未安装 `yt-dlp`，或平台拒绝下载，后端会把任务标记为 `failed`，并把错误转换成面向用户的提示。常见情况包括：

- 平台暂不支持自动下载。
- 视频需要登录、Cookie 或权限。
- 视频已删除、不可用或链接无法访问。
- 下载超时。
- 平台返回 403 或存在防盗链限制。

失败任务会清理对应的 `storage/jobs/{job_id}/` 中间目录。已经下载成功并进入分析流程的任务，在后台任务结束后也会清理任务目录；最终历史记录使用 `storage/videos/` 和 `storage/frames/` 中的持久化文件。

## 6. 一键复制脚本

一键复制面向“爆款视频脚本复刻”，不是再次拆解源视频。它把已保存的拆解结果当成参考模板，把用户自己的商品信息当成新素材，生成一版可拍摄的新脚本。

前端入口在分析详情的模型结果卡片中。用户点击“一键复制”后，会进入独立脚本复刻页面：

1. 左侧展示源视频的爆款公式、小类、分类原因、黄金 3 秒复刻说明和分镜模板。
2. 右侧填写需要带货的商品，可输入商品名，也可上传产品图。
3. “希望突出的内容”会在用户未手动编辑时自动生成默认卖点，来源是模型对商品标题、产品品类、已有输入和可选图片的解析；接口失败时才退回本地规则。
4. 用户可以点卖点推荐标签继续补充，也可以在高级设置中填写品类、目标用户、使用场景、优惠、时长、模型和语气。
5. 点击生成前，前端会推断用户商品类目，并和参考视频 `product_classification` 比较。如果不一致，会弹窗提示不同品类复刻可能导致场景、卖点或黄金 3 秒不完全适配。
6. 用户确认后，前端提交 `multipart/form-data` 到 `POST /api/analyses/{analysis_id}/script-copy`。

后端脚本复刻 prompt 会明确约束：

- 只复制结构、节奏、钩子和转化逻辑，不照抄源视频原台词、品牌、商品、价格或具体承诺。
- 新脚本必须围绕用户产品真实信息展开；用户没提供的信息不能编造成事实。
- 如果源视频命中黄金 3 秒，新脚本第一分镜必须说明前 0-3 秒画面、字幕或口播、产品出现方式和继续观看理由。
- 输出结构包括 `copy_strategy`、`shots`、`cta_options` 和 `missing_info_questions`。

产品图处理只服务于本次生成。前端用本地 object URL 显示缩略图，移除图片、离开页面或组件卸载时释放 URL；后端把上传图片转成 data URL 随请求发给模型，不写入 `storage/`，也不进入 SQLite。

## 7. 视频拆解方式

系统不是直接把完整视频文件交给模型，而是先做结构化预处理。

### 音频拆解

后端会先把上传的视频文件拆出一条临时音频文件，再把这条音频送给 Whisper 做 ASR 转写。完整流程如下：

```text
上传的视频 mp4
→ MoviePy 读取视频文件
→ 从视频轨道里取出 audio 音频轨
→ 写成本地临时文件 audio.mp3
→ faster-whisper / stable-whisper / OpenAI Whisper 读取 audio.mp3
→ 输出带时间戳的 ASR 文本
```

其中 `audio.mp3` 不是用户上传的文件，而是后端从视频里临时提取出来的中间文件。分析完成后，后端会通过 `clean_temp()` 清理临时文件，避免下一次分析复用旧音频。

ASR 输出结果类似下面这样：

```text
0.00 --> 3.20 Holy shit! Look how big this package is!
3.92 --> 6.88 Are you kidding me? It's from ComfortBitch. Let's open it together.
12.69 --> 16.21 They were on sale and I got the kid sizes.
```

这种格式中的 `0.00 --> 3.20` 表示这句话在视频第 0.00 秒到第 3.20 秒之间出现。它保留了口播内容和时间信息，方便后续 AI 按语义段落拆分分镜。

### 抽帧逻辑

当前后端有三类抽帧动作，作用不同：

1. 临时帧预抽取  
   分析流水线开始时会先调用 `extract_frames()`，清空根目录 `frames/` 下的旧图片，并按每 2 秒一张保存为 `0000.jpg`、`0002.jpg` 这类临时帧。这个目录挂载为 `/frames`，属于运行时临时文件，不是最终历史记录。

2. 视觉关键帧采样  
   调用模型前会调用 `sample_visual_frames()`，按视频时长、拆解粒度和字幕时间戳计算时间点，直接从视频中取帧。这批图片会被压缩成 JPEG、转成 base64 data URL，并作为多模态输入发给支持图片的模型。

3. 分镜缩略图抽取  
   AI 返回 JSON 后，后端会根据每个分镜的 `evidence_timestamp` 再从原视频抽一张缩略图，保存到历史记录目录，供前端分镜卡片展示。

抽帧的意义是给大模型补充视觉证据。ASR 只能告诉模型“说了什么”，不能可靠判断画面里真实出现了什么商品、颜色、场景、文字贴纸、价格页、试穿动作或对比画面。视觉关键帧让模型在拆分镜时同时参考“口播时间轴”和“画面证据”，从而减少把背景音乐歌词误判成商品说明、把不存在的商品写进结果，或把明显视觉切换合并到同一分镜。

### 视觉关键帧采样规则

前端会把拆解粒度传给后端的 `breakdown_granularity` 字段。后端只接受三个值：`coarse`、`balanced`、`fine`，非法值会回退到 `balanced`。

采样数量不是固定值，而是由“基础均匀覆盖 + 优先关键时间点 + 最大帧数上限”共同决定。

基础均匀覆盖先按视频时长和粒度计算目标间隔：

| 粒度 | 视频时长 | 目标采样间隔 | 最大帧数 |
| --- | --- | --- | --- |
| 粗略 `coarse` | `<= 30s` | 1.5s | 30 |
| 粗略 `coarse` | `<= 90s` | 3.0s | 35 |
| 粗略 `coarse` | `> 90s` | 5.0s | 45 |
| 均衡 `balanced` | `<= 30s` | 1.0s | 40 |
| 均衡 `balanced` | `<= 90s` | 2.0s | 45 |
| 均衡 `balanced` | `> 90s` | 3.5s | 60 |
| 精细 `fine` | `<= 30s` | 0.75s | 50 |
| 精细 `fine` | `<= 90s` | 1.0s | 80 |
| 精细 `fine` | `> 90s` | 1.5s | 100 |

实际帧数计算方式是：

```text
target_count = ceil(video_duration / target_step)
frame_count = min(max_frames, target_count)
```

取帧时间点不是每段正中间，而是落在每个均分片段约 45% 的位置：

```text
timestamp = index * step + step * 0.45
```

如果有 ASR 字幕或手动字幕，系统会额外加入优先关键时间点：

- 黄金 3 秒附近：`0.35s`、`1.2s`、`2.4s`，用于判断开头钩子和首屏视觉。
- 每段字幕边界附近：开始后 `+0.2s`、结束前 `-0.2s`，用于捕捉口播语义切换附近的画面。
- 如果某段字幕持续 6 秒以上，再补一个段落中点。

优先关键时间点最多占总帧预算约 40%，其余预算继续用于全片均匀覆盖，避免字幕很密时只抽到前半段或某些口播密集区域。所有时间点会按秒数排序、去重，并限制在 `duration - 0.05s` 以内，避免取到视频尾部越界。

每张关键帧会：

- 转成 RGB。
- 等比例缩略到最大 `900 x 1600`。
- 以 JPEG `quality=82` 压缩。
- 编号为 `F01`、`F02`、`F03`。
- 记录两位小数时间点，例如 `F01 1.48s`。

发送给模型时，每张图都会配一段说明：

```text
F01，时间：1.48s。请只描述这张图中能明确看到的商品和颜色。
```

因此模型不是只收到一堆没有上下文的 base64 图片。系统会把图片和它的编号、时间点成对发送，模型可以知道 `F01` 对应视频第几秒，并在输出 JSON 时把分镜绑定到 `evidence_frame` 和 `evidence_timestamp`。

不同模型的图片传入格式不同：

- `gpt-4o` 走 GPTProto `/v1/responses`，使用 `input_text + input_image`。
- `gemini-2.5-pro` 走 GPTProto `/v1/chat/completions`，使用 `image_url`。
- `claude-sonnet-4-5-20250929` 走 GPTProto `/v1/chat/completions`，使用 openai-format File Analysis，关键帧放在 `file.file_data`。
- `qwen-turbo` 当前走 DashScope 文本接口，不接收这些图片，只能依赖 prompt 文本和字幕，视觉证据能力弱于多模态模型。

发送给 `gemini-2.5-pro` 的 `messages` 结构示意如下：

```json
[
  {
    "role": "user",
    "content": [
      {
        "type": "text",
        "text": "完整分析 prompt，包含 ASR 字幕、分类体系和 JSON 输出要求"
      },
      {
        "type": "text",
        "text": "F01，时间：1.48s。请只描述这张图中能明确看到的商品和颜色。"
      },
      {
        "type": "image_url",
        "image_url": {
          "url": "data:image/jpeg;base64,..."
        }
      },
      {
        "type": "text",
        "text": "F02，时间：4.82s。请只描述这张图中能明确看到的商品和颜色。"
      },
      {
        "type": "image_url",
        "image_url": {
          "url": "data:image/jpeg;base64,..."
        }
      }
    ]
  }
]
```

发送给 `gpt-4o` 的 `/v1/responses` 结构类似，只是字段名是 `input_text` 和 `input_image`：

```json
{
  "model": "gpt-4o",
  "input": [
    {
      "role": "user",
      "content": [
        {
          "type": "input_text",
          "text": "完整分析 prompt，包含 ASR 字幕、分类体系和 JSON 输出要求"
        },
        {
          "type": "input_text",
          "text": "F01，时间：1.48s。请只描述这张图中能明确看到的商品和颜色。"
        },
        {
          "type": "input_image",
          "image_url": "data:image/jpeg;base64,..."
        }
      ]
    }
  ]
}
```

发送给 `claude-sonnet-4-5-20250929` 时，每帧同样先给文本说明，再用 file block 传图片：

```json
{
  "type": "text",
  "text": "F01，时间：1.48s。请只描述这张图中能明确看到的商品和颜色。"
}
{
  "type": "file",
  "file": {
    "filename": "F01.jpg",
    "file_data": "data:image/jpeg;base64,..."
  }
}
```

需要注意：`evidence_frame` 和 `evidence_timestamp` 是模型按上述编号和秒数生成的结构化字段，不是数据库强制校验出来的字段。模型通常会遵守，但仍可能偶尔填错。前端展示的分镜缩略图不是直接复用传给模型的原始关键帧，而是后端根据模型返回的 `evidence_timestamp` 再从原视频截一张图。

### 分镜缩略图

AI 返回每个分镜后，系统会根据分镜里的 `evidence_timestamp` 再抽取一张图片。若模型没有给出可用的 `evidence_timestamp`，后端会退回到该分镜时间段的一个偏前中点：

```text
fallback_timestamp = start_time + max(end_time - start_time, 0.5) * 0.45
```

抽出的分镜图先写入临时 `/frames/shot_01.jpg`，保存历史记录时再复制到模型隔离目录：

```text
/storage/frames/{analysis_id}/{model}/shot_01.jpg
/storage/frames/{analysis_id}/{model}/shot_02.jpg
```

重新拆解同一个视频时，不会生成新的 `analysis_id`；后端会按模型覆盖对应的 `{analysis_id}/{model}` 图片目录，并在详情页保留不同模型的最新版本。

## 8. 分析维度

系统的 prompt 内置了三套视频分析分类体系和一套产品分类白名单，并在模型返回后做规则化后处理：

- 内部爆款模式分类体系：判断 `viral_formula` 和 `formula_subtype`。
- 卖点角度分类体系：判断 `selling_point_angle` 和 `selling_point_subtype`。
- 黄金 3 秒钩子分类体系：判断 `golden_3s_hook` 和 `golden_3s_subtype`。
- 产品分类白名单：判断 `product_classification`，用于历史视频列表筛选。

核心原则是先判断全片主导表达方式，再按该类型的高转化叙事链路拆分分镜。

### 爆款大类

1. 第一人称视角  
   适用于手持近景、POV、工具演示、清洁、个护、组装、功能测试等。

2. 开箱 / ASMR  
   适用于拆包装、取出配件、触摸材质、撕膜、摆放、质感展示等。

3. GRWM + 产品  
   适用于创作者在镜前、浴室、卧室、化妆、护肤、穿搭、整理流程中使用产品。

4. 分屏对比  
   适用于左右分屏、前后对比、有无产品对比、竞品对比、平替对比等。

5. 日常 Vlog  
   适用于晨间、家庭、通勤、运动、旅行、家务等生活流植入。

### 小类

每个大类下有固定小类：

- 第一人称视角：困境解决型、过程演示型、探索发现型、日常切片型。
- 开箱 / ASMR：开箱评测型、纯感官型、惊喜揭晓型、礼物叙事型。
- GRWM + 产品：仪式步骤型、教学穿插型、场景驱动型、情感叙事型、产品对决型。
- 分屏对比：产品对比型、方法对比型、淘汰赛型、平替对决型、前后分屏型。
- 日常 Vlog：日常碎片型、从早到晚型、主题日型。

模型不能自己发明新小类，必须从体系中选择。

### 分镜叙事标签

每个分镜还会从全局叙事步骤词库中选择标签，例如：

- 故事开场
- 用户痛点
- 场景设定
- 产品亮相
- 产品信息
- 产品特写
- 使用说明
- 功能演示
- 产品卖点
- 产品差异
- 效果对比
- 真实体验
- 使用感受
- 价格促销
- 优惠活动
- 行动号召

这些标签不代表固定模板，而是按视频真实出现内容顺序标注。

### 卖点角度

每条分镜还会标注全片或该段主要购买理由：

- 专家背书
- 展示效果
- 轻松便捷
- 制造紧迫感
- 天然安全
- 价格优势
- 贩卖生活方式

后处理会根据全片文本综合判断卖点角度。如果模型没有输出卖点角度，或把普通折扣误判成“制造紧迫感”，后端会按关键词纠偏。例如只有 `sale`、`price`、`cheap` 但没有倒计时或库存压力时，更容易归为“价格优势”。

### 黄金 3 秒钩子

黄金 3 秒只根据视频开头约 `0-3s` 或最早分镜判断。可选主类包括：

- 提问式
- 挑战式
- 秘诀/技巧
- 震撼数据
- 争议式

如果开头只是普通铺垫、产品自然出现或无法判断，后处理会把 `golden_3s_hook`、`golden_3s_subtype`、`golden_3s_reason` 清空，不强行贴标签。

### 产品分类

产品分类用于历史视频列表上方的筛选控件，不参与爆款公式判断。模型输出分镜时会同时给出：

- `product_category`：自由文本，描述画面中真实出现的商品品类，例如服装、护肤、清洁、工具、食品。
- `product_classification`：固定白名单类目，只能从下列选项中选择；不确定时输出空字符串。

固定类目如下：

```text
家居用品、厨房用品、家纺布艺、家电、女装与女士内衣、穆斯林时尚、鞋靴、美妆个护、
手机与数码、电脑办公、宠物用品、母婴用品、运动与户外、玩具和爱好、家具、五金工具、
家装建材、汽车与摩托车、时尚配件、食品饮料、保健、图书&杂志&音频、儿童时尚、
男装与男士内衣、箱包、虚拟商品、二手、收藏品、珠宝与衍生品、票务与代金券
```

后端会用 `normalize_product_classification()` 做二次归一：

1. 如果模型已经输出白名单中的精确类目，直接保留。
2. 如果模型只输出了英文或更细的自由品类，后端会按关键词映射到白名单，例如 `beauty skincare` 归为 `美妆个护`，`手机壳和充电线` 归为 `手机与数码`。
3. 如果无法命中白名单，保存为空字符串，不创建动态类目。
4. 保存主记录和模型版本时，后端会把归一后的类目写入 `analyses.product_classification` 和 `analysis_versions.product_classification`。
5. 读取旧记录时，如果数据库字段为空，后端会从已有分镜 JSON 的 `product_classification` 或 `product_category` 临时推断，便于老数据继续参与筛选。

前端历史列表默认选择“全部产品分类”，此时展示所有记录。切换到某个具体类目时，只展示 `product_classification` 等于该类目的已完成分析记录；分类为空或未识别的记录不会出现在具体类目下，只会在“全部产品分类”下可见。正在排队或分析中的任务占位记录也只在“全部产品分类”下显示。

### 分类后处理规则

模型返回后，后端会调用 `enrich_storyboard_result()` 做统一后处理。主要规则如下：

1. 字段补齐：补齐 `title`、`viral_formula`、`formula_subtype`、`visual_tactic`、`conversion_point`、`product_category`、`evidence_frame`、`evidence_timestamp`、卖点角度和黄金 3 秒字段。
2. 叙事标签归一：`title` 和 `content_tag` 必须落在全局叙事步骤词库内。模型输出了自造标签时，后端会按优惠、价格、对比、试穿、情绪等关键词推断一个标准标签。
3. 爆款类型纠偏：后端会按规则顺序覆盖明显误判：
   - 包裹/包装/取出/材质/试穿/评价为主，且没有明确竞品或 dupe 对照时，纠为 `开箱 / ASMR / 开箱评测型`。
   - 同款商品不同颜色、姿态或同屏展示，没有使用前后和价格替代证据时，纠为 `分屏对比 / 产品对比型`。
   - 护肤、护发、美妆、洗护产品在浴室或镜前完成涂抹、清洗、按摩、梳理、使用后效果展示时，纠为 `GRWM + 产品 / 仪式步骤型`。
   - 同时出现明确对比关系和便宜、平替、dupe、alternative、成本等替代价值证据时，纠为 `分屏对比 / 平替对决型`。
   - 镜前 GRWM POV 单品试穿，按亮相、多角度转身、版型/尺码/购买信息展开时，纠为 `GRWM + 产品 / 仪式步骤型`。
   - 护理类内容围绕使用频率、操作步骤、成分功效和使用后体验展开时，也纠为 `GRWM + 产品 / 仪式步骤型`。
4. 卖点和钩子归一：卖点角度按全片综合文本推断；黄金 3 秒只看开头文本，未命中则清空。

## 9. 为什么这样分析

### 先分类，再拆分

短视频的转化逻辑和视频类型强相关。例如开箱视频的重点是期待感和质感，对比视频的重点是差异证明，日常 Vlog 的重点是生活场景代入。如果不先判断视频主导表达方式，分镜容易只变成普通摘要，无法沉淀可复用的方法论。

### 同时看字幕和画面

短视频里经常存在背景音乐、歌词、无关字幕。只看音频可能误判商品或卖点；只看画面又可能漏掉口播里的价格、优惠、购买理由。因此系统同时使用：

- 字幕：理解口播、价格、卖点和行动号召。
- 关键帧：判断商品、场景、人物动作和视觉证据。

### 强制 evidence_frame

每个分镜都要求绑定 `evidence_frame` 和 `evidence_timestamp`，目的是降低 AI 编造风险。也就是说，模型描述某个商品、颜色、动作时，必须能对应到具体关键帧。

### 强制 JSON 输出

统一 JSON 结构可以让结果直接被前端展示，也方便未来做：

- 批量统计不同爆款类型占比。
- 统计常见叙事链路。
- 对比不同视频的分镜结构。
- 建立爆款案例库。
- 进一步训练或评估模型输出质量。

## 10. AI Prompt 框架

实际 prompt 在后端动态生成，核心结构如下：

```text
你是专业跨境短视频爆款模式分析师，严格输出JSON数组，不要任何多余解释。

你的任务不是固定套用某一种模式，而是先判断视频属于哪一个大类与小类，再按该类型的高转化叙事链路拆分分镜。

你还会收到按时间顺序排列的视频关键帧图片，每张图都有编号和时间点。
分类、商品品类、场景和分镜必须优先依据关键帧画面证据。
字幕或音频可能只是背景音乐歌词，不能把无关歌词当成商品说明。
每个分镜必须绑定 evidence_frame 和 evidence_timestamp。

内部爆款模式分类体系：
第一人称视角、开箱 / ASMR、GRWM + 产品、分屏对比、日常 Vlog。

分类判断规则：
先判断视频主导表达方式，而不是单句台词。
如果同时命中多个类型，选择贯穿全片最多、承担转化最强的那个。
formula_subtype 必须从对应大类的小类中选择。
拆解粒度由前端传入：粗略优先合并同商品、同场景、同转化目的；均衡按重大视觉切换和转化目的变化拆分；精细会保留价格页、文字贴纸、分屏对比、产品切换、关键动作步骤或行动号召等独立视觉切点。
分镜切分优先级是：重大视觉场景切换 > 叙事目的切换 > 时间节奏。不要逐句切分，也不要为了口播完整而合并两个明显不同的视觉画面。
同一品牌内的童码/成人码、不同尺码、不同颜色、sale 折扣、cheaper 购买技巧属于开箱 haul 或产品测评中的购买信息，不等于平替对决；只有出现两个品牌、两个产品或明确 dupe/alternative/竞品对照时，才判为平替对决型。

输出格式：
[
  {
    "start_time": 数字,
    "end_time": 数字,
    "title": "叙事角色",
    "scene_description": "画面描述",
    "script": "台词",
    "product_category": "商品品类",
    "product_classification": "固定产品分类；无法判断则为空字符串",
    "evidence_frame": "证据关键帧编号",
    "evidence_timestamp": 数字,
    "shot_type": "镜头类型",
    "content_tag": "叙事标签",
    "viral_formula": "爆款大类",
    "formula_subtype": "爆款小类",
    "category_reason": "分类原因",
    "selling_point_angle": "卖点角度",
    "selling_point_subtype": "卖点角度小类",
    "selling_point_reason": "卖点判断原因",
    "golden_3s_hook": "黄金3秒钩子主类",
    "golden_3s_subtype": "黄金3秒钩子小类",
    "golden_3s_reason": "黄金3秒判断原因",
    "visual_tactic": "视觉手法",
    "conversion_point": "转化作用"
  }
]

字幕内容：
{Whisper 转写出来的 transcript}
```

## 12. 模型展示策略

为了避免用户选择不可用模型，前端模型列表不再固定写死，而是请求后端：

```text
GET /api/model-options
```

后端会根据两个条件返回模型：

1. 后端已经实现该模型调用逻辑。
2. 当前模型有可用 key。`gemini-2.5-pro` 和 `gpt-4o` 使用项目内置 GPTProto 默认 key，其他模型主要依赖环境变量。

当前判断逻辑：

- `gemini-2.5-pro`：默认使用项目内置 GPTProto key；如果配置 `GPTPROTO_API_KEY` 或 `BRAIN_API_KEY`，则用环境变量覆盖。
- `gpt-4o`：使用同一个 GPTProto key，走 `/v1/responses`，按 `input_text + input_image` 格式传入字幕和关键帧。
- `claude-sonnet-4-5-20250929`：使用同一个 GPTProto key，走 `/v1/chat/completions`，按 File Analysis 格式传入字幕和关键帧。
- `qwen-turbo`：需要 `DASHSCOPE_KEY`。
- 文心一言：当前后端未实现完整调用，所以暂不展示。

这样可以保证产品层面只暴露“点了就能用”的模型。

## 13. 当前价值

这个工程已经具备一个内容分析闭环：

- 能上传视频。
- 能自动抽取字幕和关键帧。
- 能按内部爆款模式体系分析视频。
- 能输出结构化分镜。
- 能保存历史记录。
- 能支持重新拆解。
- 能避免未配置模型在前端误展示。

对运营或内容团队来说，它可以把“人工看视频总结经验”变成“AI 先结构化拆解，人再复核和沉淀方法论”。

## 14. 后续优化方向

1. 接入更多模型时，统一走 `model-options` 配置，避免前端重复改代码。
2. 把 prompt 拆成可配置模板，方便不同业务线调整分析口径。
3. 增加批量视频分析能力，用于爆款样本库建设。
4. 增加导出能力，例如导出 Excel、PPT 或报告。
5. 增加人工修正入口，让运营人员修正分镜标签和分类结果。
6. 对 AI 输出做更严格的 JSON Schema 校验，减少格式异常。
7. 建立评估集，对比不同模型在分类准确率、分镜颗粒度、画面忠实度上的表现。
